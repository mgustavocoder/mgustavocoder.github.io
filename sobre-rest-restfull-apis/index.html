Sobre REST e Restful APIs

REST significa REpresentational State Transfer. É um padrão de comunicação cliente/servidor apresentado por Roy Fielding em 2000 em sua disertação doutorado pela universidade da Califórnia.

Esse padrão é muito popular em aplicações que disponibilizam APIs (Application Programming Interface), geralmente trafegando dados no formato JSON atráves do protocolo HTTP.

O protocolo REST utiliza os métodos do HTTP como GET, POST, PUT, PATCH e DELETE para manipular recursos no servidor.

Utilização dos métodos HTTP em APIs REST

HTTP GET
O método GET é usado para buscar a representação de um recurso, sem modificá-lo. O GET seria o READ do CRUD, e não deve alterar o recurso de nenhuma forma. Além disso, APIs GET devem ser idempotentes, o que significa que múltiplos requests idênticos, devem produzir o mesmo resultado até que uma outra API (POST ou PUT) altere o recurso no servidor.

O resultado de sucesso esperado é o HTTP Status Code 200 (OK) com o conteúdo no body.

HTTP POST
O método POST deve ser usado para criar novos recursos, e seria o CREATE na correlação com o CRUD.

O resultado de sucesso esperado é o HTTP Status Code 201 (Created) ou 202 (Accepted) caso a ação tenha sido enfileirada.

HTTP PUT
O método PUT é utilizado para atualizar recursos existentes e caso o recurso não exista, o método pode decidir se cria ou não o recurso.

O resultado de sucesso esperado varia entre os HTTP Status Code 200 (OK), 201 (Created), 204 (No Content) ou 202 (Accepted) caso a ação tenha sido enfileirada.

HTTP PATCH
O método PATCH é responsável por atualizações parciais nos recursos, enquanto o PUT é responsável pela atualização do recurso completo.

O resultado de sucesso esperado varia entre os HTTP Status Code 200 (OK), 204 (No Content) ou 202 (Accepted) caso a ação tenha sido enfileirada.

HTTP DELETE
O método delete é responsável por deletar recursos.

O resultado de sucesso esperado varia entre os HTTP Status Code 200 (OK), 204 (No content) ou 202 (Accepted) caso a ação tenha sido enfileirada.

6 requisitos da uma API RESTful

  • Interface Uniforme
    A seguinte frase exemplifica o conceito de interface uniforme:
    Uma vez que os desenvolvedores se familiarizem com uma de suas APIs, eles devem ser capazes de conseguir utilizar suas outras APIs de maneira similar.
    É preciso definir quais recursos serão disponibilizados através da API e seguir rigorosamente esse padrão, não modificando o formato da resposta entre JSON e XML entre APIs, por exemplo.
    Um recurso não deve ser muito grande e conter muita informação em seu conteúdo, ao invés disso, deve conter links para outros recursos quando necessário.

  • Cliente Servidor
    Essencialmente, essa regra diz que frontend e backend devem poder evoluir separadamente, o que é cada vez mais um padrão de mercado.
    Backend e Frontend devem poder ser desenvolvidos separadamente e até mesmo trocados, contando que a interface entre eles não seja alterada.

  • Stateless
    Não ter estado significa que o servidor não irá armazenar nada sobre a requisição HTTP que o cliente fez. Toda requisição será tratada como uma requisição nova. Sem sessão, sem histórico.
    Se uma aplicação precisa manter estado, por exemplo a autenticação do usuário, o lado do cliente precisa manter esses dados e informá-los no request em casa solicitação.
    Nenhum contexto do cliente deve ser armazenado no lado do servidor entre requisições. O cliente é responsável por armazenar o estado da aplicação.

  • Cacheable
    No REST, devemos aplicar cache quando possível, quando um recurso não muda muito com o tempo, por exemplo. Os recursos devem se declarar cacheaveis, através dos headers da requisição HTTP.
    Uma estratégia de cache bem gerenciada reduz a carga de trabalho do backend, melhorando a escalabilidade e performance da aplicação.

  • Arquitetura em Camadas
    O estilo de sistema em camadas permite que uma arquitetura seja composta de camadas hierárquicas, restringindo o comportamento do componente de forma que cada componente não possa "ver" além da camada imediata com a qual está interagindo.

  • Código sob demanda (opcional)
    Apesar de geralmente lidarmos com JSON nas respostas de nossa API, também é permitido responder código executável para suportar parte da aplicação.
